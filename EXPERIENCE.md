# Итоги проекта MLOps (ИТМО 2026)

Этот файл содержит описание проделанной работы, обоснование выбранных решений и **полный пошаговый гайд** для тех, кто хочет повторить этот проект или построить аналогичный с нуля.

---

## 1. Что мы сделали?
В рамках проекта мы создали полный цикл жизненного цикла модели машинного обучения (ML Lifecycle). Основные компоненты:
1.  **Версионирование данных и моделей (DVC)**: Мы не храним тяжелые веса моделей или `.csv` файлы прямо в Git. Используем MinIO как удаленное хранилище.
2.  **Трекинг экспериментов (MLflow)**: Все запуски обучения, гиперпараметры и метрики логируются для сравнения моделей.
3.  **Оркестрация процессов (Airflow)**: Автоматизированный DAG для переобучения модели при обновлении данных.
4.  **Сервинг (FastAPI)**: REST API для предсказания качества вина с валидацией через Pydantic.
5.  **Автоматизация (CI/CD)**: Пайплайны GitHub Actions для проверки качества кода (линтеры) и авто-тестирования.

## 2. Пошаговый гайд по воспроизведению (Tutorial)

Следуйте этим этапам, чтобы развернуть проект самостоятельно.

### Этап 1: Подготовка инфраструктуры
1.  **Git & Venv**: Клонируйте код и настройте изоляцию.
    ```bash
    git clone <your-repo-url>
    python -m venv venv
    source venv/bin/activate  # venv\Scripts\activate для Windows
    pip install -r requirements.txt
    ```
2.  **DVC pull**: Если проект уже содержит `.dvc` файлы, загрузите данные из облака:
    ```bash
    dvc pull
    ```

### Этап 2: Настройка версионирования (DVC + MinIO)
Если вы начинаете с нуля:
1.  Инициализируйте: `dvc init`.
2.  Добавьте удаленное хранилище (MinIO/S3):
    ```bash
    dvc remote add -d myremote s3://bucket-name
    dvc remote modify myremote endpointurl http://minio-server:9000
    ```
3.  Добавьте данные: `dvc add data/wine_quality.csv`.
4.  Закоммитьте только `.dvc` файл в Git.

### Этап 3: Трекинг экспериментов (MLflow)
1.  Запустите сервер MLflow (локально или удаленно):
    ```bash
    mlflow server --backend-store-uri sqlite:///mlflow.db --default-artifact-root ./mlruns
    ```
2.  В коде обучения (`train.py`) используйте `mlflow.log_param()` и `mlflow.log_metric()`.
3.  После обучения лучшая модель сохраняется в `.pkl` и также версионируется через DVC.

### Этап 4: Разработка API (FastAPI)
1.  Создайте модель данных в `src/api/main.py` с помощью Pydantic.
2.  Реализуйте загрузку модели при старте приложения (используйте `lifespan`).
3.  Запустите сервис:
    ```bash
    uvicorn src.api.main:app --host 0.0.0.0 --port 8000
    ```

### Этап 5: Автоматизация (Airflow DAGs)
1.  Поместите ваш скрипт обучения в `dags/`.
2.  Настройте `PythonOperator` или `BashOperator` для выполнения шагов:
    - Загрузка данных (`dvc pull`).
    - Обучение модели.
    - Оценка качества.
3.  Установите расписание (например, `@daily`).

### Этап 6: Контроль качества (CI/CD)
1.  Создайте `.github/workflows/ci.yml`.
2.  Добавьте шаги:
    - `Linting` (flake8 / ruff).
    - `Tests` (pytest), включая проверку эндпоинта `/predict`.
3.  Настройте секреты (логин/пароль к MinIO) в настройках репозитория.

## 3. Зачем всё это было нужно?
Основная цель проекта — перейти от простого «обучения модели в ноутбуке» к **промышленному подходу (MLOps)**.
- **Воспроизводимость**: Любой разработчик может выполнить `dvc pull` и получить идентичные данные и модели.
- **Масштабируемость**: Использование контейнеров и оркестраторов позволяет легко деплоить решение в облако.
- **Безопасность**: Секреты хранятся в переменных окружения, а не в коде.

## 4. Чему мы научились? (Ключевые навыки)
- **Инженерная культура**: PEP 8, линтеры, чистота кода.
- **Работа с данными как с кодом**: Понимание версионности артефактов.
- **Тестирование**: Написание надежных тестов для ML-сервисов.
- **Автоматизация CI/CD**: Построение пайплайнов для предотвращения ошибок в `main` ветке.
