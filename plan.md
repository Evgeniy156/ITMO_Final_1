# План реализации финального проекта по модулю MLOps

В этом документе представлен пошаговый план разработки инфраструктуры MLOps-проекта (от версионирования и экспериментов до CI/CD). Отмечайте завершенные шаги для отслеживания прогресса.

## Этап 1. Подготовка репозитория и окружения
- [ ] Ознакомиться с требованиями (файл проекта `README` или задание)
- [ ] Создать публичный репозиторий на GitLab (или GitHub, если допустимо). Можно использовать за основу [ci-cd-example](https://gitlab.com/mlops-hub/ci-cd-example)
- [ ] Создать и настроить ветки: `main` (для продакшена) и `develop` (для разработки)
- [ ] Склонировать репозиторий локально и переключиться в ветку `develop`
- [ ] Инициализировать DVC (`dvc init`) 
- [ ] Настроить удаленное хранилище (MinIO S3) как `remote` для DVC
- [ ] Добавить/проверить, что ключи доступа к MinIO добавлены в `.env` (и `.env` добавлен в `.gitignore`!)

## Этап 2. Версионирование данных и моделей (DVC)
- [ ] Скачать предоставленный датасет о качестве вина по [ссылке](https://github.com/aniruddhachoudhury/Red-Wine-Quality/blob/master/winequality-red.csv)
- [ ] Положить датасет в директорию `data/wine_quality.csv`
- [ ] Добавить данные под контроль DVC: `dvc add data/wine_quality.csv`
- [ ] Закоммитить сгенерированный файл `data/wine_quality.csv.dvc` и `data/.gitignore` в Git
- [ ] Загрузить данные в удалённое хранилище: `dvc push`
- [ ] Убедиться, что сырые данные `.csv` не трекаются в Git

## Этап 3. Трекинг экспериментов
- [ ] Создать Python-скрипт или Jupyter Notebook для обучения моделей
- [ ] Настроить логирование с помощью MLflow (или ClearML / W&B)
- [ ] Обучить 2 разные регрессионные модели (например, `LinearRegression` и `RandomForestRegressor`)
- [ ] Провести несколько экспериментов с разными гиперпараметрами, зафиксировать `random_state`
- [ ] Сравнить метрики качества обеих моделей
- [ ] Задокументировать: сохранить лучшую модель в файл (например, `models/wine_quality_model.pkl`)
- [ ] Добавить модель под контроль DVC: `dvc add models/wine_quality_model.pkl`
- [ ] Закоммитить `.dvc` файл модели в Git и запушить артефакт в S3 (`dvc push`)

## Этап 4. Оркестрация (Airflow)
- [ ] Создать директорию `dags/`
- [ ] Написать файл конфигурации (или использовать дефолтные `Variables` Airflow) для параметров
- [ ] Создать DAG (например, `wine_pipeline_dag.py`), настроить расписание `schedule_interval='@daily'`
- [ ] Реализовать task: **Загрузка данных** (чтение данных из DVC/локально)
- [ ] Реализовать task: **Обучение модели** (с использованием выбранной выше лучшей модели и пайплайна)
- [ ] Реализовать task: **Сохранение модели** (и метаданных) обратно в DVC (или имитация этого процесса, если настроено иначе)
- [ ] Проверить корректность связей между тасками в DAG (Task1 >> Task2 >> Task3)

## Этап 5. Сервинг модели (FastAPI)
- [ ] Создать директорию `src/api/` и основной файл `main.py`
- [ ] Реализовать Data Validation класс с помощью Pydantic (`class WineFeatures(BaseModel): ...`)
- [ ] Написать эндпоинт `GET /health` (`{"status": "ok"}`)
- [ ] Написать эндпоинт `GET /model-info` (возвращает информацию о текущей модели и её метриках)
- [ ] Написать эндпоинт `POST /predict` (принимает `WineFeatures` и возвращает результат `float`)
- [ ] Интегрировать загрузку актуальной версии модели `.pkl` из локальной папки/DVC при старте API-сервиса (API должно скачивать актуальный файл из MinIO/DVC, если его нет)

## Этап 6. Автоматизация тестирования и CI/CD
- [ ] Создать директорию `tests/` и написать unit-тесты для API (используя `pytest` и `TestClient`)
- [ ] Создать файл `.gitlab-ci.yml` (или эквивалент) в корне репозитория
- [ ] Настроить в CI/CD переменные окружения (Variables) для авторизации DVC в MinIO S3
- [ ] Описать pipeline Stage 1: **Проверка кода линтером** (flake8, black или ruff)
- [ ] Описать pipeline Stage 2: **Проверка доступности данных и модели в DVC** (команда `dvc pull`)
- [ ] Описать pipeline Stage 3: **Запуск тестов API** (`pytest tests/`)
- [ ] Настроить правила запуска CI: только при открытии Merge Request в ветку `main`

## Этап 7. Документация и финальная сдача
- [ ] Оформить файл `README.md`
  - [ ] Добавить структуру проекта
  - [ ] Описать процесс и результаты экспериментов с моделями (сравнительная таблица, метрики, обоснование лучшей)
  - [ ] Добавить инструкции по локальному запуску и настройке всех компонентов
- [ ] Провести финальный код-ревью: отсутствие случайных ключей или `pkl`/`csv` файлов в коммитах (чистая история)
- [ ] Сделать пуш всех изменений: `git push origin develop`
- [ ] Создать Merge Request из ветки `develop` в `main`
- [ ] Убедиться, что CI/CD pipeline прошел успешно (все зеленые галочки)
- [ ] Отправить проверяющему ссылку на успешный Merge Request
